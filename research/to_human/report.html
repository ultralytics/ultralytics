Stereo 3D Detection — capability & improvement

branch 001-stereo-centernet-gaps · KITTI R40, Car @ Moderate, true rotated 3D IoU · updated 2026-06-01

TL;DR

  • The task looked like it had zero capability (AP3D≈0) — but that was two eval/target bugs, not the model. The model's 2D detection & metric depth were accurate all along.
  • Fixed → real, trustworthy capability: 52.0% AP3D@0.5 (Car) with the documented recipe (replaces the docs' stale/un-reproducible 48%).
  • Cost-volume stereo depth (Phase 1): REFUTED — regressed ~2.7×, reverted.
  • MultiBin orientation (Phase 2): small but consistent win, KEPT — beats sin/cos on all 6 metrics (Car AP3D@0.5 +0.6, mean +1.5). But orientation wasn't the bottleneck (sin/cos already AOS≈AP). The real ceiling: tight-IoU vertical (height/Y) precision (BEV@0.7 25.9 ≫ 3D@0.7 10.4).

The two bugs (why AP3D was 0)

  1. 3D IoU was an axis-aligned approximation of rotated boxes → inflated overlap (45°-apart identical boxes scored IoU 1.0; true 0.707). Replaced with exact rotated BEV×height IoU. da9eece4
  2. Dimension priors keyed by name, looked up by int id → every lookup missed → garbage size targets → model learned them → decoder produced ~10.8 m cars (GT ~4 m) → IoU<0.5 → AP3D=0. Fixed by rekeying name→int; decoded length round-trips exactly. e8d5501d
